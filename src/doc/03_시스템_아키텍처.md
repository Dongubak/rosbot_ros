# 03. 시스템 아키텍처

> [!NOTE] 이 문서의 목적
> 실제 로봇 구동 시 노드들이 어떻게 연결되고 데이터가 어떻게 흐르는지 전체 그림을 이해한다.

---

## 전체 데이터 흐름

```
[MCU/STM32]
    │  (UART 또는 UDP)
    │
[micro_ros_agent]  ←────────────────────────────────────┐
    │  ROS2 토픽으로 변환                                 │
    ▼                                                    │
[rosbot_hardware_interfaces]                              │
  ├── RosbotSystem    ──→ /_motors_cmd                   │
  │      ↑                                               │
  │   /_motors_response                                  │
  └── RosbotImuSensor ←── ~/imu                         │
         │                                               │
         ▼                                               │
[Controller Manager (ros2_control)]                      │
  ├── diff_drive_controller ──→ /odometry/wheels         │
  │      ↑ /cmd_vel                                     │
  ├── imu_sensor_broadcaster ──→ /imu/data               │
  └── joint_state_broadcaster ──→ /joint_states          │
         │                                               │
         ├──────────────────────────────────────────────┘
         │
         ▼
[robot_state_publisher]
    │  URDF + joint_states → TF
    ▼
[/tf, /tf_static]  (base_link → odom → map)

         │
         ▼
[ekf_node (robot_localization)]
  입력: /odometry/wheels + /imu/data
  출력: /odometry/filtered + TF (odom → base_link)

         │
         ▼
[laser_filter_node]
  입력: /scan (RPLidar)
  출력: /scan_filtered

[joy_node] ──→ [teleop_twist_joy] ──→ /cmd_vel
```

---

## 노드 목록 (실제 하드웨어 기준)

| 노드 이름 | 패키지 | 역할 |
|-----------|--------|------|
| `controller_manager` | ros2_control | 모든 컨트롤러 관리 |
| `diff_drive_controller` | diff_drive_controller | 차동 구동 + 오도메트리 |
| `imu_sensor_broadcaster` | imu_sensor_broadcaster | IMU 데이터 퍼블리시 |
| `joint_state_broadcaster` | joint_state_broadcaster | 관절 상태 퍼블리시 |
| `robot_state_publisher` | robot_state_publisher | URDF → TF 변환 |
| `ekf_node` | robot_localization | EKF 센서 융합 |
| `laser_filter_node` | laser_filters | 레이저 박스 필터 |
| `joy_node` | joy | 조이스틱 입력 |
| `teleop_node` | teleop_twist_joy | Joy → Twist 변환 |
| `micro_ros_agent` | micro_ros_agent | MCU 통신 브릿지 |

## 주요 토픽

> [!TIP] 비유로 이해하기
> 토픽은 사내 방송 채널과 같다.
> 발행자(Publisher)는 방송을 내보내고, 구독자(Subscriber)는 관심 채널만 듣는다.
> 발행자와 구독자는 서로의 존재를 몰라도 된다.

### 입력 토픽 (명령)

| 토픽 | 메시지 타입 | 설명 |
|------|------------|------|
| `/cmd_vel` | `geometry_msgs/Twist` | 선속도(m/s) + 각속도(rad/s) 명령 |

### 출력 토픽 (센서 데이터)

| 토픽 | 메시지 타입 | 설명 |
|------|------------|------|
| `/odometry/wheels` | `nav_msgs/Odometry` | 바퀴 오도메트리 (원시) |
| `/odometry/filtered` | `nav_msgs/Odometry` | EKF 융합 오도메트리 |
| `/imu/data` | `sensor_msgs/Imu` | 처리된 IMU 데이터 |
| `/scan` | `sensor_msgs/LaserScan` | 원시 레이저 스캔 |
| `/scan_filtered` | `sensor_msgs/LaserScan` | 필터링된 레이저 스캔 |
| `/joint_states` | `sensor_msgs/JointState` | 모든 관절 상태 |
| `/tf` | `tf2_msgs/TFMessage` | 동적 좌표 변환 |
| `/tf_static` | `tf2_msgs/TFMessage` | 정적 좌표 변환 |
| `/battery_state` | `sensor_msgs/BatteryState` | 배터리 상태 |
| `/diagnostics` | `diagnostic_msgs/DiagnosticArray` | 시스템 진단 정보 |

### 내부 토픽 (직접 사용 X)

| 토픽 | 메시지 타입 | 설명 |
|------|------------|------|
| `/_motors_cmd` | `std_msgs/Float32MultiArray` | 하드웨어로 보내는 모터 명령 |
| `/_motors_response` | `sensor_msgs/JointState` | 하드웨어에서 받는 모터 피드백 |
| `/_imu/data_raw` | `sensor_msgs/Imu` | 원시 IMU (컨트롤러 내부용) |

> [!NOTE] `/_` 접두어 의미
> 언더스코어(`_`)로 시작하는 토픽은 내부 통신용이다.
> 일반적으로 외부에서 직접 구독하거나 발행하지 않는다.

## TF (Transform) 트리

> [!TIP] TF 비유
> TF 트리는 "신체 관절도"와 같다.
> base_link가 몸통이면, 바퀴는 다리, 센서는 눈과 귀에 해당한다.
> 모든 좌표는 이 트리를 따라 변환된다.

```
map (최종 위치)
└── odom (드리프트 누적 기준)
    └── base_link (로봇 본체 중심)
        ├── body_link (섀시)
        │   ├── cover_link
        │   └── imu_link
        ├── fl_wheel_link (앞 왼쪽 바퀴)
        ├── fr_wheel_link (앞 오른쪽 바퀴)
        ├── rl_wheel_link (뒤 왼쪽 바퀴)
        ├── rr_wheel_link (뒤 오른쪽 바퀴)
        ├── [lidar_link] (LiDAR)
        ├── [camera_link] (카메라)
        └── [range_0~3] (ToF 거리 센서)
```

### 각 프레임의 역할

| 프레임 | 발행 주체 | 설명 |
|--------|-----------|------|
| `map → odom` | EKF 노드 (옵션) | 전역 위치 (SLAM/AMCL 사용 시) |
| `odom → base_link` | EKF 노드 | 주행 기준 위치 |
| `base_link → 바퀴` | robot_state_publisher | URDF + joint_states 기반 |
| `base_link → 센서` | robot_state_publisher | URDF 고정 변환 |

## 하드웨어 vs 시뮬레이션 차이

| 구성요소 | 하드웨어 | 시뮬레이션 |
|---------|---------|-----------|
| 하드웨어 인터페이스 | RosbotSystem (C++) | gz_ros2_control 플러그인 |
| MCU 통신 | micro_ros_agent (UART/UDP) | 없음 (Gazebo가 대체) |
| 센서 데이터 | 실제 하드웨어 | Gazebo 시뮬레이션 |
| 시간 기준 | 시스템 시계 | use_sim_time=True |
| 센서 브릿지 | 없음 | ros_gz_bridge |

## 네임스페이스 지원

멀티 로봇 환경을 위해 전체 시스템을 네임스페이스로 격리할 수 있다.

```bash
ros2 launch rosbot_bringup bringup.launch.py namespace:=robot1
```

단, `tf`, `tf_static`, `diagnostics` 토픽은 네임스페이스에 포함되지 않도록
리매핑이 설정되어 있다 (TF는 전역 공유가 필요하기 때문).
